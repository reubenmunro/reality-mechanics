# Reality Mechanics Delivery Plan

## Mission

Reality Mechanics exists to increase structural perception.

Every task, investigation, implementation and feature must contribute to that mission.

---

# Delivery Principle

Every Cursor commission must finish with a tangible deliverable before context or tokens are exhausted.

Accepted deliverables include:
- Evidence report
- Characterisation
- Architectural paper
- Atlas refinement
- Practice refinement
- Calculus refinement
- Theory refinement
- Proof refinement
- Website implementation
- Calibration implementation

Never end a session with only discussion.

---

# Continuous Delivery Cycle

1. Detect structural pressure.
2. Commission a narrowly scoped investigation.
3. Produce a persisted deliverable.
4. Architect review.
5. Accept, reject or defer.
6. If accepted, update:
   - Atlas
   - Practice
   - Calculus
   - Theory
   - Proof
   - Calibration
   - Observatory
7. Repeat.

---

# Programme

Epoch I — Recover structural answers (Atlas)

Epoch II — Recover structural questions (Practice)

Epoch III — Recover order-preserving operations (Calculus)

Epoch IV — Build the structural observatory (Website)

Epoch V — Continuous calibration and stewardship

---

# Observatory Deliverables

The website must progressively reveal:
- dependency
- carrying
- retrace
- structural pressure
- ambiguity
- drift
- stability
- calibration
- structural density
- hidden structure humans cannot otherwise perceive.

---

# AI Worker Loop

Cursor's permanent responsibility is:

Observe → Investigate → Characterise → Review → Refine → Regenerate → Calibrate → Repeat.

The repository should improve continuously without losing retraceability.

---

# Success Criterion

The project succeeds when independent participants can reliably perceive structural order that was previously unavailable to them.

The Atlas is one instrument.

The Calculus is one instrument.

The Observatory is one instrument.
