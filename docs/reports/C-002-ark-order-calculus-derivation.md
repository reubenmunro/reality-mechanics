# PART A — OBSERVATIONS

## A1. What the repository already means by "Order"

**Order = the standing dependency arrangement, plus its followable path.**

- `Root Order` (`Ground`): "the governing dependency arrangement of the Atlas: what must already hold before later terms can become available … also makes that arrangement followable" (`Root Order.md:44-46`). It is explicitly *not* chronology, hierarchy, or sequence (`:44`).
- Order has two reads, already paired in the Atlas:
  - `Order Trace` "follows the dependency movement **forward**"; `Retrace Practice` "checks whether that movement still holds when followed **back**" (`Order Trace.md:17,:57`).
  - Order Trace "presents the generative direction … grounded in the reverse read: each downstream condition must be able to retrace the prior conditions … The forward path opens what may follow; the return path distinguishes confirmed dependency from projected sequence" (`Order Trace.md:39`).

So **Order is what must be preserved**: the arrangement of "what must already hold," followable forward and confirmable backward.

## A2. What the repository already means by "Ark"

There is **no bare `Ark` term**; the concept is grounded in two entries that agree:

- **`Ship and Ark`** (third order, `grounded: true`): "An ark is ordered primarily by **what must be carried** … In ark orientation, arrival matters because **something reality-bearing survives the crossing**" (`Ship and Ark.md:37`). Ark = **preservation-oriented carrying**, contrasted with Ship = destination-oriented carrying.
- **`Ark Run`** (practice, `working`): "the runtime practice by which pressure is carried through retrace, check, determination, and step **without losing dependency order**" (`Ark Run.md:38`); "bounded carrying through dependency order: pressure moves only as far as trace, scope, and warrant can hold" (`:42`).

So **Ark is the movement that carries a condition forward while preserving what must not be lost** — carrying disciplined by preservation.

## A3. The repository already places Ark and Order together

This is the decisive observation: the Atlas itself lists Ark and Order side by side as kinds of generative order (`Order Generation.md:77-78`, repeated `Order Trace.md:166-167`):

```text
Atlas     dependency order: relation holds, order carries, trace places
Ark Run   bounded carrying: pressure moves only as far as trace can hold
```

`Order Generation` `pairs [[Order Trace]]` and `carries [[Ark Run]]` (`Order Generation.md:18,:30`). The shared read is stated: "a bounded generative rule produces an ordered world" (`:101`). Order is the *ordered world*; Ark is the *bounded carrying* that moves through it.

## A4. The Constitution already defines the Calculus as this relation

The governing definition of Calculus **is** an order-preserving transformation:

- "The Calculus derives and justifies operations **and transformations**. It is concerned with **what occurs between states, not merely the names of states**" (`CONSTITUTION.md:182-184`, C004).
- "Every accepted **transformation** should either **preserve the order it carries** into its next readable form, or explicitly identify where that preservation remains unresolved" (`CONSTITUTION.md:42`).

Read against A1–A3: the "transformation between states" is **Ark** (carrying/movement); "the order it carries" and "the names of states" is **Order** (the arrangement). **The Constitution already describes the Calculus as Ark preserving Order — without naming the pair.**

## A5. The two poles are the two directions of one held connection

The relation is not two independent things but one connection read two ways (grounding from C-C001B, `Connection.md:31`):

- `Carry` = "the forward availability of held connection"; `Trace` = "the backward availability of the **same** connection" (`Carry.md:15`, `Trace.md:15`).
- **Ark is Carry enacted** (forward carrying of pressure); **Order is Trace preserved** (backward dependency that must still hold).
- "Carry and trace do not make two connections; they are directional availability within **one** connection" (`Connection.md:31`).

So Ark and Order are **forward and backward reads of one movement**: Ark opens what follows; Order confirms what must already hold. This matches Order Trace's own forward/return structure (`Order Trace.md:39`).

## A6. The failure mode and repair are already named

The relation has a named failure and a named recovery, which a Calculus needs:

- **Degenerate** = "carrying continues while the generative trace is lost" (`Order Trace.md:154-158`) — **Ark without Order** (movement that drops what must be preserved; the "Ship" over-orientation).
- **Regenerate** = "return to generative conditions from within the carrying" (`Order Trace.md:155`, `Order Generation.md:112`) — **Ark restoring Order** (carrying returns to what must hold).

This gives the candidate a validity criterion and a correction path, consistent with constitutional corrigibility (`CONSTITUTION.md` C013).

# PART B — DERIVATION (RECOMMENDATION)

## B1. The candidate Calculus

Derived from A1–A6, using only existing terms:

> **The Reality Mechanics Calculus is the cooperative relation `Order : Ark`** — the standing dependency arrangement (`Order`) and its preservation-oriented carrying (`Ark`) — **in which Ark carries a condition forward while Order is preserved through retrace.**
>
> An operation is a valid Calculus operation **exactly where its forward carrying (Ark) can be retraced to what must already hold (Order)** — i.e. where `Carry` and `Trace` remain two directions of one held connection (`Connection.md:31`).

Stated as the two poles and their cooperation:

| Pole | Atlas grounding | Role in the Calculus |
|------|-----------------|----------------------|
| **Order** | `Root Order`, `Order Trace`, `Retrace Practice` | What must be preserved: the arrangement and its retraceable path |
| **Ark** | `Ship and Ark`, `Ark Run` | The movement that carries a condition forward while preserving it |
| **`Order : Ark`** (the Calculus) | `CONSTITUTION.md:42, :182-184`; `Order Generation.md:77-78` | A transformation (Ark) that preserves the order it carries (Order) |

## B2. Why neither pole alone is the Calculus (removing concepts, not adding)

- **Order alone is not the Calculus.** Order alone is the *Atlas* — "the names of states," a static arrangement. C004 explicitly says the Calculus is "not merely the names of states" (`CONSTITUTION.md:184`). Order without movement generates nothing.
- **Ark alone is not the Calculus.** Ark alone is destination-led carrying that can lose what it carries — the **Ship**, ending in **Degenerate** (`Ship and Ark.md:37`, `Order Trace.md:154`). Movement without preserved order is drift.
- **The Calculus is the relation, not a pole.** It is the *mutual* condition: Ark gives Order forward realisation; Order gives Ark backward constraint. This **removes** the need for a separate list of stages or operators — the single relation "carry-that-preserves-order" generates them.

## B3. The operation form (what happens at each step)

An Ark step over Order is one operation. Its structure is already the `Ark Run` runtime, now read as *Order-preserving*:

```text
Order (what must already hold)
   ↑ retrace (Trace / Retrace Practice)      ← preservation check
Pressure → Trace → Check → Determine → Step   ← Ark, the carrying movement
   ↓ carry (Carry / Order Trace forward)      ← generative realisation
Order (next readable form, order preserved)
```

- Each `→` is valid **iff** the carried condition can still be retraced to what must already hold (`CONSTITUTION.md:42`; `Order Trace.md:39`; `Check.md:72`).
- `Check` is where preservation is tested "from both directions" (`Check.md:39`); `Determine` is where carrying is bounded into an answerable hold (`Determine.md:35`); `Step` opens the next posture without claiming finality (`Ark Run.md:60,96`).

This is the C-001 result restated at Calculus grain: the Practice runtime **is** an Ark carrying pressure across Order.

## B4. What the candidate unifies (evidence it is the right relation)

- **The operation node/edge split (C-C000A / C-C001B).** Order = the standing terms/arrangement (the "nodes"); Ark = the carrying transformation (the "edges"). The Calculus is precisely their *relation*, so "operation-as-state" and "operation-as-transformation" are reconciled as Order-pole and Ark-pole of one relation — exactly the repository's own "one connection, two directions" (`Connection.md:31`).
- **The Practice (C-001).** `Ark Run` (Pressure→Trace→Check→Determine→Step) is one instance of Ark; the Atlas dependency spine is Order; the Practice preserves order through the run.
- **The first-order spine.** `Boundary → Availability → Strain → Bearing → Resolution` (`Order Trace.md:88`) is Order as arrangement; carrying a live pressure along it is Ark.

## B5. Competing hypotheses considered

- **H1 (recommended): Calculus = `Order : Ark`** (the cooperative relation). Strongest — every element grounded; the Constitution already defines the Calculus this way (A4); the Atlas already pairs the poles (A3). **Confidence: MEDIUM-HIGH** as a *candidate* (it is a synthesis of grounded parts, not yet an accepted derivation).
- **H2: Calculus = Ark alone** (the runtime is the calculus). Rejected: contradicts C004 ("not merely … states" requires the Order pole) and collapses into Degenerate without Order (A2, A6).
- **H3: Calculus = Order alone** (dependency arrangement). Rejected: that is the Atlas, not the Calculus (A4; C003 Atlas authority).
- **H4: the pair is `Order Trace : Retrace Practice`** (forward/backward within Order). Rejected as *the* Calculus: both are order-reads without the live *carrying of pressure*; Ark adds the force and boundedness (`Ark Run.md:71`) that a transformation needs. Order Trace is the path; Ark is the carriage on it.

## B6. Confidence and unresolved questions

**HIGH (grounded facts):** Order = preserved arrangement (`Root Order`); Ark = preservation-carrying (`Ship and Ark`, `Ark Run`); the two are paired in `Order Generation`; the Constitution defines Calculus as order-preserving transformation; Carry/Trace are one connection's two directions.

**MEDIUM-HIGH:** that `Order : Ark` is *the* candidate Calculus (synthesis of the above).

**Unresolved (must be resolved before acceptance — not done here):**
1. **Naming.** There is no bare `Ark` term; the concept spans `Ship and Ark` (regime) and `Ark Run` (runtime). Whether to (a) let "Ark" name the preservation-carrying evidenced by both, or (b) use `Ark Run` as the operation and `Ship and Ark` as its discipline, must be decided **without inventing a new term** (C004A).
2. **Grain.** Order spans Ground→Higher; `Ark Run` is practice-grain. Whether the Calculus is stated at first-order (`Boundary→…→Resolution`) or practice-order (`Ark Run`) is the C-001 H-A/H-B question, still open.
3. **Validity sufficiency.** Whether "forward carry retraceable to what must hold" fully captures "preserve order," or whether `Check`/`Scope` must be named in the validity condition too (C-F001 flagged `Check`'s double load).
4. **`:` acceptance.** `Order : Ark` uses `:` as a named operator; per C010/C004A it cannot be promoted before independent review. This report supplies the derivation; it does not accept the notation.
5. **Directionality of the pair.** Whether `Order : Ark` or `Ark : Order` is the correct reading (generative forward vs reading backward, `Root Order.md:105-114`) is undetermined; the Atlas holds both directions of one connection, so the ordering of the pair is itself a derivation choice.

## B7. Bottom line

The cooperative relation is derivable and already latent in the repository: **the Calculus is Ark carrying a condition forward while Order is preserved through retrace** — one connection read as forward carry (Ark) and backward dependency (Order), failing as Degenerate and repaired as Regenerate. The Constitution already defines the Calculus in these exact terms (`:42`, C004); the Atlas already pairs the poles (`Order Generation`). The recommendation is to carry `Order : Ark` forward as **the candidate Reality Mechanics Calculus for testing** — resolving B6.1–B6.5 first. This report **proposes; it does not finalise or accept** (C007, C004A).
