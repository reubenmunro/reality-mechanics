## Architectural question

**Observatory demonstrates structure.**

**Pulse demonstrates behaviour through time.**

Does the Calibration runtime already satisfy this?

**Answer: Yes — for its temporal pass.**

The **Calibrated Approximation** cardiogram on `calibration.realitymechanics.nz` is a living runtime that exposes Drift, Strain, Threshold, Pulse, and Carried Strain as observable behaviour over time. It is not an article, not a dashboard, and not a graph of Atlas content. It runs a mechanism in front of the participant.

The page also hosts **Calibration Engine v1**, a separate structural instrument (retrace under perturbation). That pass demonstrates calibration **in structure**, not primarily **in time**. Both belong under Calibration today; only the cardiogram fully satisfies the Pulse commission’s temporal definition.

### Pass 1 — Calibrated Approximation (temporal Pulse)

**What is simulated**

A drifting **target** and an imperfect **approximation** that never fully tracks it.

| Variable | Role |
|---|---|
| `target` | Moves continuously via Brownian walk + slow sine (`driftTarget`) |
| `approx` | Lagging estimate; jumps only on pulse |
| `walk`, `seed`, `clock` | Drift texture and timing |

**What is measured (computed from the simulation)**

| Quantity | Definition |
|---|---|
| **Open Strain** | `\|target − approx\|` — live gap |
| **Threshold** | Constant `14` — strain that triggers a pulse |
| **Pulses** | Count of discrete corrections fired |
| **Trend** | Average strain after last 10 pulses |
| **Local average (EMA)** | Mechanism’s own recent baseline — not an imported schedule |
| **Carried Strain** | Residual gap after each pulse — explicitly surfaced, not reset to zero |

**What changes over time**

- Target drifts every frame while running.
- Strain rises until it crosses threshold.
- Pulse fires: approximation moves by a random fraction of the gap (0.3–0.85) plus noise — sometimes overshoots, undershoots.
- Carried Strain becomes the starting condition for the next climb.
- Scrolling cardiogram buffer (9 s window) and pulse history (≤ 40 values) update continuously.

**What is observed**

- SVG trace: strain line, EMA, threshold line, pulse flash, carried label.
- Readouts: Open Strain, Pulses, Trend, Threshold, Carried Strain.
- Proof cards: Drift · Pulse · Strain (one-line each).

**What is carried**

- **Carried Strain** — structural leftover after Resolution-like correction; measurably shapes the next threshold approach.
- Open Strain **never reaches zero** by construction: target keeps moving; each pulse closes only part of the gap.

**What persists**

- **Session only.** Reset or reload clears all state. No cross-visit memory (unlike Observatory’s `observatory.lastFocusId`).

**Mechanism chain (documented in `member/README.md`)**

```text
Target drifts → Strain builds → Threshold crosses → Pulse fires → Approximation corrects → Carried Strain remains
```

**Atlas grounding (documented, not enforced in code)**

| Runtime label | Atlas read |
|---|---|
| Open Strain | **Strain** — available distinction under contrast |
| Pre-pulse sustained rise | **Bearing** — strain held, not yet resolved |
| Pulse (mechanism) | **Resolution** — discrete, order-terminal correction |
| Carried Strain | Requirement that something survives Resolution |

Invariants in `member/test/invariants.test.mjs` lock this grammar: threshold-gated pulses, carried value from post-correction residual, strain never settles to zero, no participant text input.

### Pass 2 — Calibration Engine v1 (structural calibration)

**What is simulated**

Removal of one `needs` edge from a bounded in-memory read-model slice (`ENGINE_TERMS` in `calibration-engine.mjs`), then restore. No Atlas mutation.

**What is measured**

Whether affected terms can still **retrace to Relation** after perturbation. Forest states: `healthy`, `drift` (progress without retrace), `stasis` (retrace without discovery).

**What changes over time**

UI walks eight phases at 900 ms intervals after `GET /api/calibration/engine`. Each API call is stateless — recomputed from static slice.

**What is observed**

Observation · Evidence · Recommendation blocks; forest path animation; load-bearing / weakened / unresolved characterisation.

**What is carried**

Metaphorically: Ark role (test · perturb · carry · restore · observe). Not the cardiogram’s Carried Strain metric.

**What persists**

Nothing. Steward Accept / Reject / Defer buttons record intent in UI only — no backend write.

**Note on “Drift”**

On this pass, **Drift** means *progress without retrace* (structural). On the cardiogram, **Drift** means *target movement in time*. Same word, two instruments — both valid within Calibration, but only the cardiogram matches Pulse-as-time.

## Recommendation

### Primary

**Yes. Calibration is already the first Pulse instrument.**

Specifically, the **Calibrated Approximation** cardiogram is Pulse in the commission’s sense: observable runtime behaviour through time, with Drift, Strain, Threshold, Pulse, and Carried Strain as living readouts — not articles.

**Elevate, do not duplicate:**

- **Pulse** — public collection of living instruments (nav label already assigned, D-021.1).
- **Calibration** — first instrument inside Pulse; retains its name and runtime unchanged.
- **Calibration Engine v1** — second instrument on the same page today; structural retrace under perturbation. Keep under Pulse; distinguish from temporal Pulse pass in future steward copy if needed.

No new runtime is required to establish Pulse identity. D-021.1 already routed Pulse to this worker. The architectural work remaining is **naming alignment and collection framing** — not mechanism invention.

### Naming tension (document only — not implemented here)

| Layer | Today | Recommended direction |
|---|---|---|
| Public nav | Pulse | Keep |
| Page brand / title | Calibration | Pulse doorway; Calibration as instrument heading |
| Mission pillar | “Calibration measures the quality of participation” | Still accurate — Pulse is the surface, Calibration is the measure |
| Submission 001 | “Calibration” | Update when Proof copy is next revised |

Smallest architecture that explains the existing runtime:

```text
Pulse (surface — behaviour through time)
 └── Calibration (instrument 1 — temporal cardiogram)
 └── Calibration Engine v1 (instrument 2 — structural retrace test)
 └── [future instruments]
```

### What not to do

- Do not rebuild the cardiogram as a new Pulse page.
- Do not merge Observatory and Pulse — structure vs time are complementary instruments.
- Do not embed Theory or Atlas prose on Pulse — compression without loss means translation, not duplication.

## Future instrument model

Pulse becomes a **collection of living instruments** — each an observable runtime behaviour, not a dashboard.

| Future instrument | Likely domain | Status |
|---|---|---|
| **Calibration** | Drift → Strain → Threshold → Pulse → Carried Strain | **Live** (cardiogram) |
| **Calibration Engine v1** | Retrace under perturbation | **Live** (structural) |
| Rhythm | Timing, interval, memoryless vs patterned pulse | Not built |
| Trace | Backward retrace as lived behaviour | Partially in Field movement traces; not a Pulse instrument |
| Carry | Forward availability under hold | Partially in Field rendering; not a Pulse instrument |
| Recovery | Restore after perturbation | Ark role in engine; not standalone |
| Participation | Quality of engagement over time | Mission language; not yet instrumented |

Each instrument should follow Calibration’s contract:

- Mechanical, not AI.
- No Atlas mutation from runtime.
- No participant text input required.
- Observable behaviour dominates; prose orients only.
- Steward decides; instrument recommends.

Observatory remains the place where **structure** becomes visible through participation in the Field. Pulse remains the place where **behaviour through time** becomes visible through mechanisms that run.
