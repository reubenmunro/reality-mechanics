# D-021.4 — Pulse Instrument Contract

**Programme:** Public Structure  
**Type:** Pulse Instrument Contract  
**Mode:** Investigation only — no implementation  
**Date:** 2026-07-06 (UTC+12)  
**Context:** D-021.3 established Pulse as the public surface for observable behaviour through time; Calibration as its first instrument.

---

## Objective

Identify what makes Calibration a Pulse instrument, separate that from Calibration-specific mechanics, and produce the smallest contract future Pulse instruments can satisfy.

No implementation. No redesign. No Atlas or Theory edits.

---

## Investigation summary

Calibration hosts **two instruments** on one page today:

| Instrument | Mechanism | Primary grain |
|---|---|---|
| **Calibrated Approximation** | Drifting target, threshold-gated pulse, carried residual | **Behaviour through time** (client `requestAnimationFrame`) |
| **Calibration Engine v1** | Perturb → retrace test → restore on read-model slice | **Behaviour through steps** (stateless API + phased UI) |

Pulse identity (D-021.3) attaches primarily to the **temporal cardiogram**. Engine v1 already satisfies most Pulse **invariants** but not every **candidate clause** — which is why falsification matters.

Evidence sources: `member/src/index.js`, `member/src/calibration-engine.mjs`, `member/test/invariants.test.mjs`, `member/README.md`, D-021.3.

## Candidate contract — falsification

| Candidate clause | Verdict | Compressed form |
|---|---|---|
| Demonstrates a mechanism | **Survives** | Required |
| Changes through time | **Rejected as stated** | Engine v1 unfolds by **steps**, not continuous time. → **Behaviour unfolds** (continuous or phased) |
| Can be observed without explanation | **Rejected as stated** | Calibration includes orientation copy (“Reasoning has mechanics”). → **Observable without Theory**; minimal orientation permitted |
| Produces measurable behaviour | **Survives, narrowed** | → **Surfaces readable quantities or states** derived from the mechanism (numeric or categorical) |
| Carries state | **Rejected as universal** | Engine API is stateless; cardiogram state is session-local. → **Maintains mechanism state for the observation session** |
| Has persistence appropriate to its domain | **Rejected as requirement** | Calibration persists **nothing** cross-visit. → **Persistence is explicit, bounded, and optional** |
| Remains mechanical rather than conversational | **Survives** | Required — no AI inference as instrument core |
| Never mutates Atlas directly | **Survives** | Required — aligns with Runtime Principle P1 |
| Provides recommendation rather than authority | **Rejected as universal** | Cardiogram **demonstrates only** — no recommendation. → **Interpretive instruments separate observation from recommendation; demonstrative instruments may omit recommendation** |

## 1. Surviving contract

The smallest contract that survives falsification against Calibration:

> A **Pulse instrument** is a **mechanical, live runtime** that **unfolds a named mechanism** so a participant can **observe readable behaviour** without opening Theory. It **translates** structure where needed but **does not mutate Atlas**, **does not decide for the steward**, and **does not require conversational input** unless a future commission explicitly permits it.

### Eight clauses (normative)

| # | Clause | Meaning |
|---|---|---|
| **C1** | **Mechanical** | No AI inference, chat completion, or autonomous agent as the instrument core |
| **C2** | **Unfolding** | Behaviour changes over an observation session — continuously (`rAF`) or through declared phases |
| **C3** | **Observable** | Mechanism and outputs are visible in the UI; Theory is not required to watch |
| **C4** | **Readable** | Surfaces quantities or states computed from the mechanism (numbers, traces, labels, phases) |
| **C5** | **Session-scoped** | Mechanism state lives in the observation session; cross-visit persistence is optional and must be declared |
| **C6** | **Non-mutating** | No writes to Atlas source, D1 canon, Garden, or other authority stores from instrument runtime |
| **C7** | **Non-authoritative** | Instrument demonstrates or recommends; steward decides; no Calculus promotion from runtime |
| **C8** | **Bounded input** | Control inputs only (start, pause, reset, walk, select preset) unless a commission explicitly authorises free-text participation |

### Orientation rule (non-normative but expected)

Short instrument-level copy may orient the participant (“what am I watching?”). It must not duplicate Theory or embed Atlas articles. One screen of orientation is sufficient; the mechanism must dominate.

## 2. Rejected clauses

| Rejected | Reason |
|---|---|
| Must change through **continuous** time | Engine v1 is phased; future instruments may be event-driven |
| Observable **without any** explanation | Calibration orients before Start; rejection of Theory ≠ rejection of all prose |
| Must **carry** state between events | “Carry” is an Atlas term; session state suffices |
| Must **persist** across visits | Calibration deliberately stateless; Observatory already uses optional persistence elsewhere |
| Must **recommend** | Demonstrative instruments (cardiogram) recommend nothing |
| Must use **cardiogram / strain grammar** | Calibration-specific |
| Must **read live Atlas/D1** | Engine uses local slice; valid translation pattern |
| Must be **temporal-only** | Structural step instruments may live under Pulse when they unfold observably |

These should be enforceable the way `member/test/invariants.test.mjs` enforces Calibration today:

| Invariant | Testable signal |

| Type | Recommendation | Steward UI |
|---|---|---|
| **Demonstrative** (cardiogram) | None | None required |
| **Interpretive** (engine v1) | Observation / Evidence / Recommendation | Accept / Reject / Defer (intent only, no backend write) |

## 4. Instrument lifecycle

```text
Orient → Activate → Unfold → Observe → [Read] → [Recommend] → [Steward] → Release
```

| Phase | Participant action | Instrument duty |
|---|---|---|
| **Orient** | Arrive at Pulse surface | Minimal copy: what mechanism runs |
| **Activate** | Start / Walk / Reset | Mechanism begins from declared initial conditions |
| **Unfold** | Watch | Behaviour changes over time or steps |
| **Observe** | Read trace, readouts, phases | Outputs stay legible without Theory |
| **Read** (optional) | Follow Atlas link external to instrument | Instrument does not embed Atlas body |
| **Recommend** (optional) | Read recommendation block | Only interpretive instruments |
| **Steward** (optional) | Accept / Reject / Defer | Records intent only; no canon write |
| **Release** | Pause / Reset / leave | Session state clears or persists per declared domain |

Calibration cardiogram: Orient → Activate → Unfold → Observe → Release.

Calibration Engine v1: Orient → Activate → Unfold → Observe → Recommend → Steward → Release.

Pulse prepares the participant to feel mechanics **before** structural observation in the Field. The surfaces must not merge; contracts differ.

## Architectural chain

```text
Pulse          public surface — collection of instruments
  ↓
Instrument     named runnable unit (Calibration, Rhythm, …)
  ↓
Mechanism      the simulated or tested behaviour (cardiogram loop, perturb walk)
  ↓
Observation    what the participant sees (trace, readouts, phases)
  ↓
Reading        optional connection to Atlas vocabulary (external, steward-mediated)
```

**Theory sits outside this chain.** It explains the discipline; it is not a layer below Reading.

## 9. Future instrument assessment

Assessment against the **eight-clause contract** (C1–C8). No implementation implied.

| Instrument | Fit | Notes |
|---|---|---|
| **Calibration** (cardiogram) | **Full** | Reference demonstrative instrument; defines temporal Pulse |
| **Calibration Engine v1** | **Full** | Reference interpretive instrument; phased unfold, not continuous |
| **Rhythm** | **Strong** | Natural extension of pulse timing, interval, memoryless vs patterned spacing — same C1–C8, new mechanism |
| **Carry** | **Moderate** | Forward residual after correction — cardiogram already exposes Carried Strain; standalone instrument would isolate “what survives” as temporal behaviour, not Field relation rendering |
| **Trace** | **Moderate** | Backward retrace as phased behaviour fits C2; Field already traces movement spatially — Pulse Trace would be temporal/sequential, must not duplicate Observatory |
| **Recovery** | **Moderate** | Restore-after-perturbation is an engine phase today; standalone instrument fits interpretive pattern (C7, recommendation optional) |
| **Participation** | **Conditional** | Mission language points here, but **watch-only default (C8) conflicts** with measuring participant engagement. Requires **explicit contract amendment** before any free-text or identity-bound input |

|---|---|---|
| Structure graph | **Observatory** | Spatial, not behavioural |
| Atlas editing | **Garden / stewardship** | Mutates canon |
| Programme evidence | **Proof** | Static package |
| Why discipline works | **Theory** | Prose authority |

## 10. Recommendation

### Adopt the eight-clause contract (C1–C8) as the Pulse instrument standard.

It is the smallest set that:

- Explains why Calibration qualifies (both passes),
- Rejects over-generalisation from Calibration-specific grammar,
- Leaves room for Rhythm, Carry, Trace, and Recovery,
- Flags Participation as requiring a separate commission before input surfaces.

### Do not codify in runtime yet

This commission is architectural. When an instrument is implemented:

1. Declare **demonstrative** or **interpretive** type.
2. Map mechanism → observations explicitly.
3. Add invariants to `member/test/` (or successor) mirroring Calibration.
4. Keep Pulse nav as collection doorway; instruments remain named sections or routes under it.

### Naming follow-through (from D-021.3, not implemented here)

Pulse = surface. Calibration = first instrument name inside it. Future instruments receive their own names under the same contract — not sub-brands of Calibration.
